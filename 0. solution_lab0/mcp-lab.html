Build your first MCP server.
Learn the three building blocks.

MCP has three types of capabilities. Each one answers a simple question: who decides when it happens? Once you know the answer, you know which type to use.

You'll build one server with a mortgage example, adding tools, resources, and a prompt step by step. By the end you'll have a working server and understand how to build any MCP server.

Setup & scaffold

MCP solves a real problem: you write your server once, and any AI tool can use it. No custom code for Claude, or ChatGPT, or the next model. The server tells each client what capabilities it has; the client uses them.

We'll use FastMCP — a Python library that removes all the boring protocol code. And we'll use uv to run Python directly without fussing with virtual environments.

1 · Create a project folder and open it in VS Code

Create a new empty folder for this project — for example mcp-lab — somewhere on your machine. Then open it in VS Code: File → Open Folder… and select that folder. Everything you create during this lab (server.py, .vscode/mcp.json) lives inside this one folder. Do not use the folder that contains this lab file.

2 · Install uv

Open the integrated terminal in VS Code: Terminal → New Terminal (or Ctrl+`). Then run the command for your OS:

terminal · install uv (once)
# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh
# Windows (PowerShell)
# powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

Verify the install before continuing:

terminal
uv --version

You should see something like uv 0.x.x. If the command is not found, close and reopen the terminal (or restart VS Code) so the updated PATH takes effect.

3 · Create server.py

In VS Code's Explorer panel (Ctrl+Shift+E), click the New File icon (or right-click in the panel → New File) and name it server.py. Paste the scaffold below. The first four lines are inline metadata that tell uv which libraries to install. uv run server.py handles installation automatically — no virtual environment needed.

server.py
# /// script
# requires-python = ">=3.10"
# dependencies = ["fastmcp"]
# ///
from fastmcp import FastMCP

# The name is what clients show in their UI.
mcp = FastMCP("rabo-mortgage-advisor")

# Synthetic sample data. In a real server this would come from
# an internal rate service. We expose it as a RESOURCE later.
MORTGAGE_RATES = {
    "annuity 10 year fixed": 3.65,
    "annuity 20 year fixed": 4.10,
    "annuity 30 year fixed": 4.35,
    "linear 20 year fixed": 4.05,
    "variable rate": 4.80,
}
MAX_DTI_RATIO = 0.30  # max share of gross income for housing

if __name__ == "__main__":
    mcp.run()  # stdio transport — ideal for a first, local server

4 · Test-run the server

Before registering anything, run the server once manually to confirm there are no errors:

terminal
uv run server.py

You should see output like Starting MCP server… without any errors. Press Ctrl+C to stop it. If it crashes here, fix it before continuing — it will fail the same way when VS Code tries to start it.

5 · Register the server with VS Code

You don't start the server manually during use. Instead, you register it once in a config file and VS Code starts it automatically when needed.

In VS Code's Explorer panel, click the New Folder icon and name it .vscode. Then inside that folder, create a new file named mcp.json. Together they form the path .vscode/mcp.json. Paste this content:

.vscode/mcp.json
{
  "servers": {
    "mortgage-advisor": {
      "type": "stdio",
      "command": "uv",
      "args": ["run", "server.py"]
    }
  }
}
Tip · find the exact path to server.py

"server.py" is a relative path that works as long as you opened the correct project folder in VS Code. If VS Code can't find the file, get the full absolute path by running this in the terminal:

terminal · Windows (PowerShell)
(Resolve-Path server.py).Path
terminal · macOS / Linux
realpath server.py

Copy the output — for example C:\Users\yourname\mcp-lab\server.py — and paste it into the args array. Use forward slashes in the JSON, even on Windows:

.vscode/mcp.json · with absolute path
{
  "servers": {
    "mortgage-advisor": {
      "type": "stdio",
      "command": "uv",
      "args": ["run", "C:/Users/yourname/mcp-lab/server.py"]
    }
  }
}

Alternatively, "${workspaceFolder}/server.py" is a VS Code variable that always resolves to the open project folder — no path lookup needed.

After saving mcp.json, a Start button appears inline at the very top of the file (look just above the opening {). Click it. VS Code starts the server process and connects to it.

Open Copilot Chat with Ctrl+Alt+I (Windows / Linux) or ⌃⌘I (Mac), or click the chat icon in the Activity Bar on the left. Your server will appear under MCP Servers in the chat panel.

▸Why connect VS Code before writing any capability?

When VS Code connects, it asks your server what it can do. Starting from an empty-but-running server means every tool, resource, or prompt you add shows up the moment you reload — you always know immediately whether a problem is in your code or in the connection.

▸Prefer a real project over a single file?

The inline metadata keeps this lab to one self-contained file. When it grows into something you maintain, switch to a uv project so the dependencies live in pyproject.toml and a lockfile:

terminal
uv init rabo-mcp-lab
cd rabo-mcp-lab
uv add fastmcp
uv run server.py

Inside a project you can drop the inline header — uv run uses the project's locked environment instead.

01Model-controlled

Your first tool

Why this is a tool

A tool is a function the AI model can call by itself. The model reads your description (the docstring) and decides: "Do I need this? Should I call it now?" The docstring is not a comment — it's the instruction manual the model uses to understand what your function does.

Add a tool that calculates a monthly mortgage payment. That's it: a decorator (@mcp.tool), type hints, and a clear explanation. The model will understand what it does and when to call it.

server.py · add below the sample data
@mcp.tool
def calculate_monthly_payment(
    principal: float,
    annual_rate_pct: float,
    term_years: int,
) -> dict:
    """Calculate the monthly payment for an annuity mortgage.

    Args:
        principal: Loan amount in euros.
        annual_rate_pct: Nominal annual interest rate, e.g. 3.65.
        term_years: Term of the loan in years.

    Returns:
        Monthly payment, total amount paid and total interest.
    """
    monthly_rate = annual_rate_pct / 100 / 12
    n = term_years * 12

    if monthly_rate == 0:
        monthly_payment = principal / n
    else:
        monthly_payment = principal * monthly_rate / (1 - (1 + monthly_rate) ** -n)

    total_paid = monthly_payment * n
    return {
        "monthly_payment": round(monthly_payment, 2),
        "total_paid": round(total_paid, 2),
        "total_interest": round(total_paid - principal, 2),
    }
✓
Checkpoint

In Copilot Chat, click the Tools button. You should see calculate_monthly_payment listed. Ask Copilot: "Calculate the monthly payment for a €300,000 loan at 4.1% over 20 years." It will call your tool and return the result.

▸What did FastMCP do behind the scenes?

FastMCP turned your function signature and docstring into instructions the model understands. If you write principal: float, the model knows it's a number. If you say "Loan amount in euros", the model knows what it means. Write clear names and descriptions, and the model will call your tool correctly. Write vague ones, and it will call it wrong.

02Model-controlled

A second tool — and why you don't orchestrate

Why add a second one

A server has many tools, and the model can call them in any order it wants. You don't tell the model which tool to call next — you just provide good tools with clear descriptions, and the model figures out the sequence on its own.

server.py · add after the first tool
@mcp.tool
def affordability_check(
    monthly_payment: float,
    gross_monthly_income: float,
) -> dict:
    """Check whether a monthly payment fits a debt-to-income limit.

    Args:
        monthly_payment: Proposed monthly housing cost in euros.
        gross_monthly_income: Gross monthly income in euros.

    Returns:
        The debt-to-income ratio and whether it stays within the limit.
    """
    ratio = monthly_payment / gross_monthly_income
    return {
        "dti_ratio": round(ratio, 3),
        "max_allowed": MAX_DTI_RATIO,
        "within_limit": ratio <= MAX_DTI_RATIO,
    }
✓
Checkpoint

In Copilot Chat, ask: "Can someone earning €5,000/month afford €380,000 over 30 years at 4.35%?" Watch Copilot call calculate_monthly_payment first, then affordability_check — by itself, in order, without you telling it which tool to use.

▸Where a side-effecting tool would live

Both tools here are pure computations. A tool that changes something — logging an advice request to an internal system, say — would sit right alongside them. That side-effecting nature is exactly what separates a tool from a resource, and it's where audit logging and access control attach in production. Hold that thought for Step 05.

03App-controlled

Resources — read-only context

Why this is a resource, not a tool

Think of a policy document at a real bank. Nobody asks the model to "decide whether to look up the lending rules" — you hand it the document before it starts. That's exactly what a resource is: you choose when to make it available; the model just reads it. A tool is something the model calls on its own initiative. A resource is background context you load.

Add a single resource that exposes the bank's lending policy as a readable document — products, current rates, and eligibility rules all in one place. An advisor attaches it at the start of a conversation and the model has everything it needs before it writes a single word.

server.py · add after the tools
@mcp.resource("docs://lending-policy")
def lending_policy() -> str:
    """Bank lending policy: products, rates, and eligibility rules (synthetic).

    Include this resource as context before advising a client.
    The model reads it — you decide when to make it available.
    """
    return (
        "MORTGAGE LENDING POLICY (synthetic data)\n"
        "=========================================\n\n"
        "Current products and rates:\n"
        "  annuity 10 year fixed : 3.65%\n"
        "  annuity 20 year fixed : 4.10%\n"
        "  annuity 30 year fixed : 4.35%\n"
        "  linear 20 year fixed  : 4.05%\n"
        "  variable rate         : 4.80%\n\n"
        "Eligibility rules:\n"
        "  Max debt-to-income ratio : 30% of gross monthly income\n"
        "  Max loan-to-value        : 100% first-time buyers, 90% otherwise\n"
        "  NHG guarantee up to      : €435,000\n"
        "  Max age at end of term   : 75 years\n"
        "  Stress test rate         : 5.0%\n\n"
        "Product guide:\n"
        "  Annuity  – equal monthly payments for the full term\n"
        "  Linear   – decreasing payments (higher at start, lower at end)\n"
        "  Variable – payment moves with the market rate"
    )
✓
Checkpoint

In Copilot Chat, click the paperclip (Attach Context) icon → MCP Resource → select docs://lending-policy. Then ask: "What mortgage products are available and what are the current rates?" The model reads the policy document and answers — no tool call, no function execution.

▸Why is this obviously not a tool?

Imagine asking: "Should the model decide whether to read the bank's policy manual?" Of course not — you always want it to have the manual. Making it a tool would mean the model first has to decide: "Do I need to look this up?" That wastes a round-trip and might get skipped entirely. A resource is context you attach. The model reads it. Done.

04User-controlled

The prompt — where it all comes together

Why add a prompt?

A prompt is a workflow a person starts deliberately, like a slash-command in a chat app. It's where you tie tools and resources together into one repeatable task. You say: "First read the rate, then calculate the payment, then check affordability, then give advice." The model follows that sequence every time. Without the prompt, the model might skip a step or do things in the wrong order.

→ step 4: the model writes it up
server.py · add after the resources
@mcp.prompt
def mortgage_advice(
    client_name: str,
    loan_amount: float,
    term_years: int,
    gross_monthly_income: float,
    product: str,
) -> str:
    """Guided mortgage-advice workflow for an advisor."""
    return (
        f"You are a mortgage advisor preparing advice for {client_name}.\n\n"
        f"Follow these steps:\n"
        f"1. Read the resource `docs://lending-policy` to get the rate for '{product}' and the eligibility rules.\n"
        f"2. Call `calculate_monthly_payment` with principal={loan_amount}, "
        f"the rate from step 1, and term_years={term_years}.\n"
        f"3. Call `affordability_check` with that payment and "
        f"gross_monthly_income={gross_monthly_income}.\n"
        f"4. Write a short, structured advice: product and rate, monthly "
        f"payment, total interest, and whether it is affordable. If it is "
        f"not within the limit, suggest one alternative.\n\n"
        f"Keep the tone factual. All figures are indicative."
    )
✓
Checkpoint

In Copilot Chat, type /mortgage_advice and fill in the parameters (name, loan amount, term, income, product). Copilot reads the rate, calls both tools, and writes the advice — one trigger, same steps every time.

▸Prompt arguments are strings under the hood

The MCP spec passes prompt arguments as strings. FastMCP lets you annotate them with real types (float, int) for a better developer experience and coerces them for you. The return value can be a plain string, as here, or a list of structured messages when you need finer control over roles.

Reflect & ship

You now have a working server with all three types. When you build the next one, just ask one question about each capability: who decides when it happens? Is it the model? The app? Or the user? Your answer is your choice.

Who decides when it happens?It's aUse this when
The AI model
(on its own initiative)		ToolYou want the model to call a function whenever it makes sense
The application
(you decide what to load)		ResourceYou want to give the model read-only context, like data or a manual

(they click a button)		PromptYou want a repeatable workflow that starts with a click or slash-command

Why this design matters

The three types aren't just theory. Each one has different security and audit rules:

Tools — can change things, so you log every call. If the tool deletes data or sends money, you need a record.
Resources — are just data you hand over. Your only concern is: should this model see this data? Is it private?
Prompts — are workflows you control. You can version them, test them, and know exactly what they do.

Running your server for real

Right now you ran it over stdio — direct pipe to the client. For multiple people to connect, switch to HTTP:

server.py
if __name__ == "__main__":
    mcp.run(transport="http")  # HTTP for multiple clients

Over HTTP, you can add login (OAuth), track who called what, and manage permissions. That's the next level.

Next steps after this lab

You've learned the three building blocks. When you're ready to go deeper: Tasks let your server run long operations in the background. Sampling lets you use the client's AI model instead of your own. And HTTP transport lets multiple clients connect. One step at a time.
