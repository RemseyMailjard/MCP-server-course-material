Demo Lab

In this lab you extend the design demo into a more useful MCP server. You add a richer component, expand the token system, document composition rules, and build a better preview flow so Copilot can return design output that is easier to review and reuse.

1. Overview

The current server already returns component bundles and a small token set. In this lab you make that output more useful for real development work: add a richer component, make theming more explicit, document how components should be combined, and return previews that show the result in context.

A

Component depth

Add a richer component with nested structure, state, and responsive behaviour.

B

Token system

Expand the theme set so the server can return more than one visual direction.

C

Documentation

Add guidance for composition, accessibility, or motion so outputs stay consistent.

D

Previewing

Make it easy to assemble the final design bundle into a polished preview page.

What this server should help you do

The goal is to move from a vague UI request to a concrete, reusable implementation. That means finding the right component, understanding its structure, applying the correct token set, reading the relevant guidance, and returning a preview page that is useful for review or handoff.

Example prompt What it should return
Search for the best component for a launch story and show me the outline. A ranked component match with structure notes and a recommended follow-up prompt.
Give me the same component in a sunset theme and explain the design choice. HTML, CSS, and JS for the component plus the selected token set and a short explanation.
Build a story page for the feature panel and include composition guidance. A polished preview page that combines the component, tokens, outline, and documentation topic.

2. Learning goal

By the end of the lab you should understand how this server models components, documentation, themes, and preview helpers as separate MCP outputs. The final result should not only work, but also show a clear design decision that Copilot can explain and reuse.

Expected learning outcomes

  • Add a new component that is more complex than a single button or card.
  • Extend the design tokens with an additional theme or variant.
  • Create a documentation topic that teaches composition, hierarchy, or accessibility.
  • Build a preview helper that combines a component with supporting context.
  • Validate the extension through MCP prompts and the returned HTML/CSS/JS bundles.
Keep the scope tight: the strongest solutions are small, coherent, and clearly designed, not crowded with unrelated components or styles.

3. What you extend

Start with the existing catalog, then add one deeper layer at a time: a new component, a wider token set, better guidance, and a more complete preview flow. That mirrors how a design system grows in practice.

4. Lab flow

Work from the content outward. Add the design decision first, then give it data, then improve the preview and the prompt experience.

Inspect the current design language Review the existing component names, the token set, and the documentation topics so the new work aligns with the current demo style.
Add one richer component Choose a component that has multiple parts, such as a feature panel, hero block, or callout module with supporting text and actions.
Expand the token set Add a theme variant that changes the visual mood enough to prove the preview page is driven by tokens rather than hardcoded colors.
Add documentation for composition Create a topic that explains when to use the component, how to combine it with other pieces, and what accessibility rules matter.
Build and validate the preview flow Return a polished page or story view so Copilot can evaluate the component in context, not just as isolated snippets.

Concrete build targets

Tool What the developer should build Done when
search_components Find the right component by query terms like launch, hero, panel, navbar, or action. The tool returns a ranked match list with a next-step hint.
get_component_outline Expose the component structure in a short, readable format for agents. The response includes parts, notes, and a suggested prompt.
build_component_page Return a complete preview page that respects the selected theme and outline. The HTML renders with chips, notes, and the component markup.
build_component_story_page Combine a component, a theme, and a documentation topic into a richer review page. The page shows the prompt, the outline, the guidance, and the preview.

5. Build directions

Complete the first three items as the core lab. Use the fourth if you finish early or want a stronger end result.

Build A: Feature panel component

Add a richer component, such as a feature panel or hero block, that includes an eyebrow, title, supporting text, and one or more actions.

Goal: make it discoverable through search_components, document its structure with get_component_outline, and expose it through get_component_bundle.

Build B: Theme expansion

Add a second or third token theme so the server can generate a different visual identity for the same component.

Goal: update get_design_tokens with a clearly distinct palette and shadow treatment.

Build C: Composition guidance

Add documentation for how the component should be combined with other blocks, including spacing, hierarchy, and focus states.

Goal: extend get_design_documentation with a new topic such as composition or motion.

Build D: Story page builder

Create a new helper that turns a component, theme, and documentation topic into one polished preview page for demos or handoff.

Goal: return a complete HTML shell through build_component_story_page that wraps the component and explains the design choices.

6. Implementation tasks

Treat these as concrete implementation slices. Each task should end in a usable MCP response that Copilot can call and reuse immediately.

Task 1: Add a new component

Add a new component entry in COMPONENTS and include a richer HTML structure, component-scoped CSS, and a small JavaScript enhancement.

Expected developer output: a feature panel with eyebrow, body, primary CTA, and supporting link.

"feature-panel": {
  "name": "Feature Panel",
  "purpose": "Showcase a product or workflow with supporting actions.",
  "variant": "composed",
  "html": "...",
  "css": "...",
  "js": "...",
  "notes": ["Use on product pages.", "Pair with a short action label."]
}

Task 2: Add a new documentation topic

Expand DOCUMENTATION_TOPICS with a topic that explains composition rules or motion guidance.

Expected developer output: a topic that helps Copilot explain how the feature panel and navbar work together.

"composition": {
  "title": "Composition Guidance",
  "summary": "Use the demo components together without losing hierarchy.",
  "guidelines": [
    "Keep one primary action.",
    "Balance text and whitespace.",
    "Maintain visible focus states."
  ],
  "example_prompt": "Explain how to combine a hero and a button component."
}

Task 3: Expand the token system

Add a new token theme in get_design_tokens so the same component can ship in a different visual mood.

Expected developer output: at least one new theme that changes background, accents, and shadow style.

"sunset": {
  "background": "#fff7ed",
  "surface": "#ffffff",
  "text": "#1f2937",
  "muted": "#6b7280",
  "primary": "#c2410c",
  "secondary": "#fb923c",
  "radius": "1rem",
  "shadow": "0 22px 45px rgba(194, 65, 12, 0.18)"
}

Task 4: Build a story page helper

Add a new tool that combines a component bundle with a selected theme and doc topic into one clean preview page.

Expected developer output: a story page that shows the prompt, the outline, the guidance, and the rendered component.

@mcp.tool()
def build_component_story_page(component_id: str, theme: str = "ocean", topic: str = "overview") -> dict[str, str]:
    """Return a complete design story page with component, tokens, and guidance."""
    ...
Design note: avoid adding random extra components. The best lab submissions show a clear design system decision, not a larger pile of snippets.

9. Success criteria

1

Clear design intent

The new work feels like part of one design system, not a collection of unrelated snippets.

2

Good layering

Components, docs, tokens, and preview helpers each do one job and do it cleanly.

3

Useful output

Copilot can use the new bundle in a way that is ready for review, not just for display.

Validation checklist

  • The new component has a meaningful structure and responsive CSS.
  • The token theme changes the visual tone without breaking readability.
  • The documentation topic gives practical guidance instead of generic advice.
  • The preview helper makes the component easier to evaluate in context.
  • The lab can be run from the folder without extra setup guesses.
